# Daily Journal – 7 July 2026

## Objective

Perform an in-depth technical evaluation of multiple approaches for extracting content from JavaScript-rendered websites and identify the most suitable solution for integration into the C++ web crawler.

## Research and Evaluation

### 1. Backend API Extraction

The first approach explored was extracting data directly from a website's ``backend APIs`` by analyzing network traffic through browser developer tools.

A detailed investigation revealed that modern web applications generate a large number of network requests, including requests initiated by the browser itself, third-party services, analytics providers, advertisements, and the application's backend APIs. Distinguishing application-specific API endpoints from browser-generated and third-party requests proved to be highly challenging.

Because of the complexity involved in reliably identifying the correct API endpoints for arbitrary websites, this approach was considered impractical for building a generic web crawler and was therefore rejected.

---

### 2. Browser Automation Using Puppeteer

The second approach involved implementing browser automation using **Puppeteer**.

A complete prototype was developed using Node.js. The implementation successfully launched a Chromium instance, navigated to the target webpage, waited for JavaScript execution to complete, and extracted the fully rendered HTML content. From a functional perspective, the approach worked as expected and was capable of crawling modern JavaScript-driven websites.

However, Puppeteer has a mandatory dependency on the **Node.js runtime**. Since the project requirements restricted the implementation to C++ without relying on a Node.js environment, this solution did not satisfy the architectural constraints.

Although technically successful, the Puppeteer-based implementation was discarded due to its runtime dependency on Node.js.

---

### 3. Chromium Embedded Framework (CEF)

The third approach investigated was **Chromium Embedded Framework (CEF)**.

A detailed study showed that CEF embeds an entire Chromium browser into a native C++ application, providing complete browser functionality including JavaScript execution, rendering, networking, and browser APIs.

While CEF fully satisfies the functional requirements, it introduces significant overhead in terms of binary size, runtime memory consumption (approximately 500–600 MB), and implementation complexity. Integrating CEF would effectively require maintaining an embedded browser environment, making it unsuitable for a lightweight web crawler.

Considering the resource requirements and increased project complexity, this approach was also rejected.

---

### 4. Research and Implementation Using Chrome DevTools Protocol (CDP)

After evaluating the previous approaches, the research shifted toward the **Chrome DevTools Protocol (CDP)**.

A detailed study was conducted to understand the protocol architecture, browser targets, debugging sessions, and the communication model based on WebSockets and JSON-RPC messages.

A proof-of-concept implementation was developed in C++ by launching a headless Chromium instance with the remote debugging interface enabled. The implementation established a WebSocket connection with the browser's DevTools endpoint and communicated directly with Chromium using CDP commands.

The workflow consisted of:

* Launching Chromium in headless mode with remote debugging enabled.
* Discovering the active DevTools target.
* Establishing a WebSocket session with the browser.
* Sending CDP commands to navigate to a webpage.
* Waiting for page loading and JavaScript execution to complete.
* Retrieving the fully rendered HTML using DOM-related CDP commands.
* Downloading the rendered page content without requiring Node.js or an embedded browser framework.

This approach provided direct access to Chromium's rendering engine while maintaining a lightweight native C++ architecture. It successfully eliminated the dependency on Node.js and avoided the heavy runtime overhead associated with Chromium Embedded Framework.

## Outcome

After an extensive technical evaluation of multiple approaches, **Chrome DevTools Protocol (CDP)** was identified as the most suitable solution for the web crawler. It provides access to fully rendered webpage content through Chromium's native debugging interface while satisfying the project's architectural constraints of remaining a lightweight, C++-based implementation without external browser automation frameworks or embedded browser runtimes.
